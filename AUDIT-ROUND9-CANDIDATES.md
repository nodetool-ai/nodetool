## #1 [HIGH] security — packages/agents/src/tools/control-tool.ts:83
**Control tool ignores its declared property whitelist — arbitrary node-property injection into controlled nodes**

- Description: ControlNodeTool builds jsonSchema from the target node's control_actions.run.properties (the whitelist of properties the agent is allowed to override) but never enforces it: the schema omits additionalProperties:false (control-tool.ts:83-87). In the StepExecutor path, control-tool calls bypass process()/schema validation entirely — runTool intercepts `tool instanceof ControlNodeTool` and calls createControlEvent(cleanArgs) directly (step-executor.ts:953-956), where cleanArgs is the raw model-supplied args minus _message. createControlEvent stuffs all of them into the RunEvent as `properties` (control-tool.ts:109-111). In actor.ts `_runControlled`, these properties are merged LAST — `{...node.properties, ...dynamic_properties, ...inputs, ...event.properties}` (actor.ts:1383-1388) — so they override every real property of the controlled node, including properties the node author never exposed as a control action. An agent (or a prompt-injected agent via the view_image/image-injection vector or malicious tool output) can therefore set arbitrary properties on any downstream controlled node.
- Scenario: A workflow gives an agent a control edge to an HTTP request node whose control_actions.run.properties only declares {body}. The agent calls the control tool with {body:"...", url:"http://169.254.169.254/latest/meta-data/"}. No validation strips `url`; the RunEvent carries properties {body, url}; actor.ts spreads them over the node's configured url, so the node fetches the attacker-chosen internal URL (SSRF). The same mechanism can rewrite a file-write node's path, a code node's code, or an LLM node's system prompt.
- Suggested fix: Enforce the whitelist: set additionalProperties:false in ControlNodeTool.jsonSchema, and in createControlEvent() (and the step-executor interception) filter args down to the keys declared in control_actions.run.properties before constructing the RunEvent, dropping/ignoring any undeclared property.

## #2 [HIGH] resource-leak — packages/runtime/src/context.ts:1634
**HTTP error responses thrown without draining/cancelling the body — undici socket leak on every 4xx/final 5xx**

- Description: httpRequestWithRetries cancels the unconsumed response body only on the retry path (line 1626, with a comment explicitly explaining that under undici an unconsumed body keeps the socket checked out of the connection pool). But both error-exit paths skip the cancel: (1) the non-retryable !response.ok branch (lines 1630-1638) throws immediately with the body unread, and (2) a retryable status (e.g. 503) on the FINAL attempt fails the `attempt < maxRetries - 1` guard at line 1616, falls through to the same throw, again without cancelling. The file's own asset-resolution fallback (resolveAssetBytes, lines 2102-2152) deliberately probes multiple /api/storage and /api/assets URLs expecting 404 misses, so this leak fires on routine asset lookups, not just error cases.
- Scenario: A workflow resolves assets via the HTTP fallback (no local storage hit). Each probe returns 404, downloadFile -> httpGet throws 'HTTP 404', and the Response body is never consumed or cancelled. Under Node's undici fetch each such response pins a socket/stream out of the keep-alive pool until GC finalization. A long-running server processing many workflows accumulates pinned sockets and 'body not consumed' resource churn, eventually exhausting file descriptors / pool capacity and stalling all outbound HTTP.
- Suggested fix: Before throwing the nonRetryable error (and on the final-attempt retryable-status path), drain the body: `await response.body?.cancel().catch(() => {})` — same as the retry branch. Optionally read a bounded amount of the body into the error message first.

**resolveUri allows arbitrary local file read via file:// scheme and /api/storage/ path traversal**

- Description: resolveUri() is called with user-controlled URIs taken straight from chat message content (anthropic-provider.ts:316, openai-provider.ts:653/1100/1140, ollama-provider.ts:264, kie-provider.ts:358, responses-api.ts:41). **collect() destructively consumes non-driver max-scope side inputs, dropping all but the first driver envelope per key**

- Description: In _runCorrelatedImpl, when a repeating driver handle exists, tryFire drains the driver backlog via `while (isReady(key))`. But collect() handles every non-list max-scope handle identically: `bucket.shift()` plus `handleBuckets.delete(key)` when empty (lines 708-711). ## #14 [MEDIUM] correctness — packages/runtime/src/providers/cassette-provider.ts:111
**stableStringify flags shared (acyclic) references as circular, corrupting the request hash**

- Description: In stableStringify's `encode`, every visited object is added to `seen` (line 111) but is NEVER removed after its subtree is encoded. `seen` is therefore a set of all-ever-visited objects, not the current DFS path. Any object that legitimately appears more than once in the request graph (a DAG, not a cycle) has its second and later occurrences replaced by the literal "[circular]" instead of its real content. ## #16 [MEDIUM] resource-leak — packages/agents/src/agent-workflow-runner.ts:122

- Description: execute() starts runner.run(...) as a detached runPromise (lines 95-110) and streams its messages through a monkey-patched context.emit.
