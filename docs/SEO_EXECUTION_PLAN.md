## Shape of the work

Three layers, ten PRs, two ops setups. Layer 1 is Next.js code in `marketing/` (pages exist after
it ships, empty of generated content). Layer 2 is generation tooling (fills the pages). Layer 3 is
CI (keeps them filling). Dependencies run strictly downward — Layer 2 output is consumed through
one interface (the generated data modules), so layers can be built by different people in
parallel after PR-1.

| # | Deliverable | Layer | Depends on | Size |
|---|---|---|---|---|
| PR-1 | Page-data infrastructure: entry types, derived sitemap, route-walking smoke tests | 1 | — | M |
| PR-2 | Template engine: `/templates/*` + generic graph renderer + hub | 1 | PR-1 | L |
| PR-3 | Showcase route: `/showcase/*` reading the generated manifest | 1 | PR-1 | M |
| PR-4 | Model pages: `/models/*` + model-vs-model routes | 1 | PR-1, W-1 output | M |
| PR-5 | Comparison factory: `/vs/*` + `/alternatives/*` from data | 1 | PR-1 | M |
| PR-6 | Landing matrix + task hubs | 1 | PR-2 | M |
| PR-7 | FAQ / ideas / glossary hubs | 1 | PR-1 | M |
| W-1 | Showcase Seeder (batch generation + manifest) | 2 | — | M |
| W-2 | Model Duel (same-prompt pairs) | 2 | W-1 | S |
| W-3 | Copy Writer (model-page draft agent) | 2 | — | S |
| W-4 | Model watcher (manifest diff) | 2 | PR-4 | S |
| OPS-1 | `seo-seed.yml` (cron + dispatch → PR) | 3 | W-1, PR-3 | S |
| OPS-2 | `model-watch.yml` (weekly) | 3 | W-4 | S |

Sizes: S ≈ half a day, M ≈ 1–2 days, L ≈ 3–4 days for one engineer familiar with the repo.

## Layer 1 — the factory (`marketing/`)

### PR-1: page-data infrastructure

The one-time plumbing every engine plugs into. No new public pages ship in this PR.

- `marketing/src/data/types.ts` — the shared entry contract:

  ```ts
  export type PageEntry = {
    route: string;            // "/templates/movie-posters"
    title: string;            // full <title> text
    description: string;      // meta description
    priority: number;         // sitemap hint
    changeFrequency: "weekly" | "monthly";
    indexable: boolean;       // false → noindex + excluded from sitemap
  };
  ```

  Each engine module (`templateEntries`, `showcaseEntries`, `modelEntries`, `competitorEntries`,
  …) exports `entries: PageEntry[]` plus its engine-specific fields.
- `marketing/src/data/registry.ts` — imports every engine module, exports the concatenated list.
- Rewrite `marketing/src/app/sitemap.ts` to map over the registry instead of its hand-ordered
  21-entry array. The existing static routes move into a `staticEntries.ts` module so nothing is
  special-cased.
- Rewrite the `ROUTES` array in `marketing/tests/e2e/smoke.spec.ts` to walk the registry
  (`entries.filter(e => e.indexable).map(e => e.route)`), keeping the existing assertions
  (status < 400, NodeTool title, exactly one `h1`). For engines with hundreds of pages, the spec
  samples: all hub pages plus the first three entries per engine.
- A `yearToken()` helper (build-time current year) for title templates.

**Acceptance**: `npm run build` in `marketing/` produces a sitemap identical to today's (same 21
URLs) — this PR is a refactor with proof of no behavior change; e2e smoke suite passes; adding a
dummy entry to a data module adds it to sitemap and smoke coverage with no other edits.

### PR-2: template engine (`/templates/*`)

- `marketing/scripts/generate-template-entries.mjs` — reads
  `packages/base-nodes/nodetool/examples/nodetool-base/*.json`, emits
  `marketing/src/data/templateEntries.generated.ts` (checked in, regenerated by an npm script,
  CI fails if stale — same pattern as other generated artifacts in the repo). Per template:
  slug, name, description, tags, node-type list with counts, graph geometry (below), and the
  matching thumbnail from `packages/base-nodes/nodetool/assets/nodetool-base/<name>.jpg`
  (42 card-art files already exist — copy referenced ones into `marketing/public/templates/`).
- **Generic graph renderer.** The flow primitives exist (`marketing/src/components/flow/`:
  `NodeCanvas`, `FlowNode`, `FlowEdge`, `NodeProperty`, `HandleRow`) but every current usage
  (`WorkflowGraph.tsx`, `MoviePosterGraph.tsx`, `MovieTrailerGraph.tsx`,
  `app/flow-preview/page.tsx`) hand-places nodes and edge anchors. The new
  `WorkflowGraphFromJson` component computes both from the workflow JSON: node positions from
  each node's `ui_properties.position`, edge anchors from source/target node geometry (fixed row
  height per property, same approach the hand-built graphs encode manually). Server-rendered,
  scaled-to-fit like `WorkflowGraph.tsx`. This component is the PR's main effort and is reused
  by PR-4 and PR-6.
- `app/templates/[slug]/page.tsx` with `generateStaticParams` from the generated module. Section
  order per the playbook: output (or thumbnail fallback) → graph → nodes list → how to run →
  related templates (8–9 by shared tags) → `HowTo` + `ItemList` JSON-LD via `JsonLd.tsx`.
- `app/templates/page.tsx` hub (category-grouped card grid), linked from the footer and from
  `/use-cases` teasers. One `opengraph-image.tsx` for the dynamic route rendering the template
  name over its card art.
- Index gate in the generator: templates with a boilerplate description (< 80 chars) or no
  thumbnail get `indexable: false`.

**Acceptance**: 37 template pages build statically; every page passes the smoke assertions; the
graph for `Movie Posters` visually matches the editor layout (manual check); Lighthouse
performance on a template page ≥ 90 (`marketing/lighthouserc.json` already runs in CI); no page
ships with an empty output *and* empty walkthrough section.

### PR-3: showcase route (`/showcase/*`)

Ships the route and manifest contract; pages appear when W-1 lands its first batch.

- `marketing/src/data/showcaseEntries.generated.ts` — one row per generation:

  ```ts
  export type ShowcaseEntry = PageEntry & {
    id: string;               // short hash, unique
    prompt: string;
    modelSlug: string;        // FK → modelEntries
    provider: string;         // "fal_ai"
    workflowSlug: string;     // FK → templateEntries
    asset: { kind: "image" | "video" | "audio"; src: string; poster?: string };
    params: Record<string, string | number>;
    batchId: string;          // provenance: which seed run produced it
  };
  ```

- `app/showcase/[slug]/page.tsx`: hero asset → prompt as H1 → model/workflow/provider chips →
  params → 9 related items (same model, same workflow, same batch — in that priority) → remix
  CTA (`https://app.nodetool.ai/...` open-with-workflow deep link; if the deep link can't
  pre-fill the prompt yet, link the workflow and file the product gap). `ImageObject`/
  `VideoObject` + `ItemList` JSON-LD.
- `/showcase` hub with model and workflow filters (static filter pages, not client-side state,
  so crawlers see them: `/showcase/model/<slug>`, `/showcase/workflow/<slug>`).
- Near-duplicate rule enforced at ingest, not render: entries carry `canonicalOf?: string`; the
  page emits the canonical link and `indexable: false`.

**Acceptance**: route builds with a fixture manifest of 5 entries committed as test data; hub
filter pages link every entry; a `canonicalOf` fixture renders the canonical tag correctly.

### PR-4: model pages (`/models/*`)

- `marketing/src/data/modelEntries.ts` — hand-written per model (this is deliberate: the
  capability blurb is the intent-match, per the playbook's node-docs lesson). Fields: slug, name,
  vendor, modality (`text-to-video` etc.), blurb (2–3 paragraphs), capability facts (max
  duration/resolution, audio), FAQ items. Provider coverage is **not** hand-written: a generator
  step greps the node-package manifests (same source `generate --list-models` reads) and emits
  `modelProviderCoverage.generated.ts` mapping model slug → provider ids.
- `app/models/[slug]/page.tsx`: showcase heroes (query `showcaseEntries` by `modelSlug`, take
  top 3) → blurb → provider table (from coverage + BYOK framing) → embedded template using the
  model (via `WorkflowGraphFromJson`) → prompt examples (more showcase items) → FAQ →
  `SoftwareApplication` + `FAQPage` JSON-LD. Title: "<Model> in NodeTool — run it in a visual AI
  workflow (<year>)".
- `app/models/[pair]/page.tsx` for `modelComparisonEntries.ts` (slug pairs + verdict copy +
  same-prompt showcase pairs from W-2, matched on `params.duelId`). `BreadcrumbList` + `FAQPage`.
- Launch set: Seedance, Veo 3, Kling, Sora, Hailuo, Wan, Seedream, FLUX, GPT-Image, Imagen
  (all verified present in `packages/*-nodes/src`), plus the ~10 highest-volume pairs.

**Acceptance**: a model page with zero showcase items still builds (thumbnail fallback, no empty
hero section); provider table matches `generate <provider> --list-models` output for two
spot-checked providers; pair pages render both outputs side by side from the fixture manifest.

### PR-5: comparison factory (`/vs/*`, `/alternatives/*`)

- Extract the six hand-built pages (`app/vs/{comfyui,weavy,langflow,n8n,flowise,dify}/page.tsx`,
  ~250 lines each) into `competitorEntries.ts` (name, category, feature-table rows, concession
  copy, FAQ items) + one `app/vs/[slug]/page.tsx`. **Content parity is the review bar**: render
  each old and new page and diff visible text before deleting the folders.
- Add `app/alternatives/[slug]/page.tsx` on the same data (different template: limitation intro,
  4–6 tool list, table, FAQ).
- `ComparisonMesh` component (in-content "Compare NodeTool with other tools" block) on both
  templates; year token in titles.
- Extend to the first wave of new competitors (Flora, Krea, LM Studio, Jan, Lindy, Gumloop) as
  data rows — copy drafted with W-3's pattern, human-edited.

**Acceptance**: text parity on the six existing pages; footer "Compare" column derives from the
data module; every comparison page links ≥ 8 siblings; new competitor pages carry at least one
honest concession row (the credibility pattern from the shipped pages).

### PR-6: landing matrix + task hubs

- `landingEntries.ts` feeding `app/solutions/[slug]/page.tsx` (or extend `useCaseEntries.ts` —
  decide in the PR; the existing `/use-cases/*` folders stay as-is either way, redirected or
  absorbed). Swappable slices: hero + featured template embed. Shared body components already
  exist as sections (`FeaturesSection`, `UseCasesShowcase`, etc. in `marketing/src/components/`).
- `app/tasks/[slug]/page.tsx` for task hubs (image-to-video, lip-sync, upscale-video,
  text-to-music…): models doing the task (from `modelEntries.modality`) + templates wiring it +
  showcase strip. `Product` + `FAQPage` JSON-LD.
- First wave: the four `/marketing` "coming soon" use cases (assets now exist via W-1), the two
  missing personas (local-first, researchers), and six task hubs.

### PR-7: FAQ / ideas / glossary

- `faqEntries.ts` (question, answer-md, category, relatedRoute) → `app/faq/[slug]/page.tsx` +
  `/faq` hub. The same rows render as FAQ blocks on model/landing/comparison pages — one source,
  two surfaces, `QAPage` on the standalone page.
- `app/ideas/[slug]/page.tsx` generated per category from `templateEntries` (needs no new data).
- Glossary: per-term anchors already exist in `docs/glossary.md` on the docs site; on the
  marketing site ship only `/faq/*` terms with search demand. Do not duplicate the docs glossary
  wholesale (duplicate-content risk between the two domains).
- Regenerate `marketing/public/llms.txt` from the registry in the same PR.

## Layer 2 — generation tooling (`marketing/seo/`)

The interface between layers is two directories: `marketing/seo/out/<batch-id>/` (assets +
`manifest.jsonl`, produced here) and `marketing/src/data/*.generated.ts` (produced by the ingest
script, consumed by Layer 1).

### W-1: Showcase Seeder

Start from the proven in-repo pattern:
`packages/base-nodes/nodetool/examples/nodetool-base/gallery_thumbnail_generator.ts` already
iterates every workflow JSON, calls OpenAI (concept) + fal (render) through
`@nodetool-ai/runtime` providers with secrets from the local store, and writes assets to a fixed
directory. The seeder is that script generalized:

- `marketing/seo/seed.ts` (run with `npx tsx`): flags `--template <slug>`, `--models <csv>`,
  `--count <n>`, `--budget-usd <n>`, `--batch <id>`. For each template × model: an LLM writes
  `count` prompt variants inside the template's domain (short system prompt per template category
  lives in `marketing/seo/prompts/<category>.md`), then the generation provider renders each.
  Every output appends a `manifest.jsonl` row matching `ShowcaseEntry` minus the derived fields.
- Budget enforcement: the runtime providers already report per-call cost
  (`gen_ai.usage.cost_usd` in `llm_call` events); the script sums and stops at the cap, writing
  the manifest up to that point. A killed batch is still a valid batch.
- Dedup at generation time: normalize prompt, hash (model, normalized prompt); skip on collision
  with existing manifest rows.
- Second step (`--as-workflow` flag, later): run the same batch through the DSL
  (`create_graph`/`run_graph`, as in the `*_dsl.py` / DSL examples) so the seeder itself is a
  publishable NodeTool workflow. Ship the script first; the dogfood version is a follow-up with
  identical manifest output.
- `marketing/scripts/ingest-showcase.mjs`: reads `manifest.jsonl` files, applies the index gate
  (near-duplicate canonicals, min prompt length, asset exists and decodes), moves assets to
  `marketing/public/showcase/<batch>/`, regenerates `showcaseEntries.generated.ts`.
- **Asset weight decision (make in this PR)**: the site deploys via `opennextjs-cloudflare`, and
  per-file/aggregate limits plus repo bloat make committed video untenable beyond the first
  batches. Default: images committed under `marketing/public/` (compressed, ≤ 300 KB), video
  uploaded to an R2 bucket behind a `media.nodetool.ai` custom domain, manifest stores absolute
  URLs. The ingest script handles both from day one so nothing moves later.

**Acceptance**: `seed.ts --template movie-posters --models flux-schnell --count 5 --budget-usd 2`
produces 5 assets + manifest locally with only `FAL_API_KEY`/`OPENAI_API_KEY` configured; ingest
turns the manifest into entries that render on the PR-3 fixture route; re-running the same seed
command adds zero new rows (dedup).

### W-2: Model Duel

Thin wrapper over W-1: `--duel <modelA>,<modelB>` runs every prompt through both models, stamps
both rows with a shared `params.duelId`. PR-4's pair pages join on it. Curated duel prompt sets
(5–8 per pair, chosen to expose real differences: motion, text rendering, physics) live in
`marketing/seo/prompts/duels/<pair>.md`.

### W-3: Copy Writer

The shipped `SEO Content Engine` example (strategist agent → briefs → Markdown per brief) with a
retargeted brief: input is a model slug + its provider-coverage rows + vendor facts pasted by the
operator; output is `marketing/seo/drafts/models/<slug>.md` containing the blurb, capability
facts, FAQ candidates, and title/meta. A human edits the draft into `modelEntries.ts` — drafts
never ship unreviewed. Run via `npm run dev:nodetool -- run` on the exported DSL, or in the
editor. Also usable for competitor-page first drafts (PR-5 wave 2).

### W-4: Model watcher

`scripts/model-watch.mjs` (plain Node, no workflow): for each provider with a manifest, run
`nodetool generate <provider> --list-models --json`, diff model ids against
`modelProviderCoverage.generated.ts` + a `watchlist.json` of ignored ids. New id → print a
machine-readable report (consumed by OPS-2). No API-key-free mode exists for every provider, so
the watcher runs where keys are configured (CI secrets or a maintainer machine).

## Layer 3 — CI glue (`.github/workflows/`)

### OPS-1: `seo-seed.yml`

- Triggers: `workflow_dispatch` with inputs (`templates`, `models`, `count`, `budget_usd`,
  `duel`) and a weekly `schedule` running a small default batch (one template rotation,
  `budget_usd: 10`).
- Steps: checkout → `nvm use` / `npm ci` → `npx tsx marketing/seo/seed.ts` with provider keys
  from repo secrets (`FAL_API_KEY`, `OPENAI_API_KEY`; add per model set) → upload video to R2
  (`wrangler r2 object put`, `CLOUDFLARE_API_TOKEN` secret) → `ingest-showcase.mjs` →
  `git checkout -b seo/seed-<batch>` → open a PR (`peter-evans/create-pull-request` or the repo's
  existing PR automation).
- The PR body embeds a review checklist: outputs eyeballed (no broken/blank/off-brand renders),
  prompts read as human, no duplicate slugs, image weight within budget, spend total vs cap.
  **Merging the PR is the index gate.** Nothing auto-merges.
- Guard: the job fails (and posts no PR) if manifest row count is 0 or spend exceeded cap by
  provider error.

### OPS-2: `model-watch.yml`

Weekly cron: run W-4; if new models are found, open one issue per provider batch with the report
and a pre-filled seed command. A maintainer turns the interesting ones into a `modelEntries.ts`
row + W-3 draft + seed batch. Deliberately not auto-PR: a new model id is a signal, not content.

## Rollout timeline and checkpoints

Assumes one engineer plus one content reviewer, part-time. Weeks are calendar, not effort.

- **Weeks 1–2**: PR-1, PR-2. First 37 template pages live. Submit updated sitemap in Search
  Console.
- **Week 3**: PR-3, W-1. First seed batch locally (2 image templates × 2 models × 10 prompts,
  ~40 pages) merged by hand to shake out the ingest path. OPS-1 lands once that merge is clean.
- **Week 4**: W-2, PR-4 with the 10 launch models + first 5 duels. PR-5 factory conversion.
- **Weeks 5–6**: PR-5 new competitors, PR-6, W-3 drafting model/competitor copy. Scale showcase
  to ~200 pages via the weekly OPS-1 batch.
- **Weeks 7–8**: PR-7, OPS-2, distribution push from SEO Strategy §6 timed to the "our site is
  built by our own workflows" post.

**Checkpoints** (GSC segmented by URL prefix, per the playbook's measurement rule):

- **Week 4**: `/templates/*` pages indexed (Coverage report). If < 50% indexed, fix crawl/quality
  before adding showcase volume — do not scale into an indexation problem.
- **Week 8**: first non-branded impressions on `/templates/*` or `/models/*`. Showcase indexation
  rate ≥ 60% of submitted. If showcase indexation is poor, halve batch size and raise the gate
  (longer prompts, fewer near-duplicates) instead of pushing volume.
- **Week 12**: clicks (not just impressions) on at least two engines; model pages ranking top-20
  on at least two "<model> + workflow/api/try" queries. Engines with zero non-branded impressions
  get their keyword axis re-examined (playbook guardrail) before any further pages.

## Open decisions (owner calls, blocking the marked items)

1. **Provider API keys as GitHub repo secrets** (blocks OPS-1/OPS-2; W-1 runs locally without
   it). Alternative: seed batches run only on a maintainer machine, CI does ingest-and-PR from an
   uploaded manifest artifact.
2. **R2 bucket + media domain** (blocks video showcase at scale; image-only batches fit in-repo).
3. **Remix deep link**: can Cloud open a workflow with a pre-filled prompt? If not, PR-3 ships
   with a plain workflow link and the deep link becomes a product ticket.
4. **`/solutions/*` vs extending `/use-cases/*`** for PR-6 (naming only, decide in the PR).
